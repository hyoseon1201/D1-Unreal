## 2. HUD 쿨타임 표시 기능

### C++ 수정
- **`FOnCooldownSignature` 확장:** `ThreeParams` → `FourParams` (`Duration` 추가)
  - 이유: Ability마다 쿨타임이 GE의 Scalable Float로 다르게 지정되어 있어, WBP Timer에서 Duration이 필요함
- **`OnCooldownTagChanged` 로직:** Cooldown GE가 여러 개일 경우 가장 긴 `TimeRemaining`을 찾도록 전체 순회 (기존 `Times[0]`만 사용하던 버그 수정)

### WBP 작업 (Blueprint)
- **구조:** `D1OverlayWidgetController` → `OnCooldownTagChangedDelegate` Broadcast → `WBP_SkillSlot` 바인딩
- **이벤트 흐름:**
  1. `AbilityInfoDelegate` 수신 시 `CurrentAbilityTag` 저장
  2. `OnCooldownTagChangedDelegate` 수신 → `CurrentAbilityTag` 필터링
  3. `Percent > 0`: `ImageCooldown` Visible + `CooldownDynamicMat.SetScalarParameterValue("Progress", Percent)` + `Set Timer by Event(0.03s, Looping)` → `UpdateCooldown` Custom Event
  4. `Percent == 0`: `Clear and Invalidate Timer` + `Progress = 0` + `ImageCooldown` Hidden
  5. `UpdateCooldown`: `ElapsedTime += 0.03` → `1.0 - (ElapsedTime / CooldownDuration)` → 머터리얼 Progress 갱신
- **머터리얼 수정:** `M_Cooldown`의 `VectorToRadialValue` 입력 X축에 `Multiply(-1, 1)` 적용하여 회전 방향 반시계 → 시계로 변경
- **트러블슈팅:**
  - `CooldownDuration`이 0으로 초기화되어 `ElapsedTime / 0` 발생 → Delegate에 `Duration` 파라미터 추가하여 해결
  - 빈 슬롯에 쿨타임 머터리얼이 계속 남아있음 → `AbilityInfoDelegate`에서 `AbilityTag == CurrentAbilityTag && InputTag != MySlotTag` 조건으로 슬롯 초기화 로직 추가

### 아키텍처 패턴
```
Ability 활성화 → Cooldown GE 적용 → Cooldown Tag 추가
    ↓
WidgetController: RegisterGameplayTagEvent → OnCooldownTagChangedDelegate.Broadcast(AbilityTag, CooldownTag, Percent, Duration)
    ↓
WBP_SkillSlot: Delegate 수신 → Timer 시작 → MI_Cooldown.SetScalarParameterValue("Progress", percent)
```
- WidgetController는 시작/종료 신호만 전달
- 시각적 갱신(Tick)은 WBP 내 Timer가 담당 (Lyra 표준 방식)

## 3. 버그 수정

### SelectedAbilityTag 초기화 문제
- **파일:** `D1SkillMenuWidgetController.cpp`
- **원인:** `EquipSkill`에서 `SelectedAbilityTag = FGameplayTag()`로 초기화하여, 장착 직후 선택 태그가 날아가 연속 장착이 불가능했음
- **해결:** 초기화 코드 제거. 사용자가 다른 아이콘 클릭 시 `AbilitySelected`가 자연스럽게 덮어씀

### 슬롯 덮어쓰기 시 이전 어빌리티 발동 문제
- **파일:** `D1AbilitySystemComponent.cpp`
- **원인:** `ClearSlot` 함수가 완전히 비어 있었고, `ServerEquipAbility`에서 기존 슬롯 태그를 제거하지 않고 덮어써서 동일 `InputTag`를 가진 Spec이 2개 이상 생김. `GetSpecFromInputTag`가 순회 시 먼저 발견된 기존 Spec을 반환하여 이전 어빌리티가 계속 발동됨
- **해결:**
  - `ClearSlot`에 기존 슬롯 검색 → `InputTag` 제거 → 상태 `Equipped`→`Unlocked` 변경 → 클라이언트 브로드캐스트 로직 구현
  - `ServerEquipAbility` 시작 부분에 `ClearSlot(SlotTag)` 호출 추가

## 4. 인벤토리 시스템 설계 확정
- **Phase 1 (언리얼 납부):** 웹서버 없이 언리얼 낶에서만 동작하는 인벤토리 시스템 구축
  - `UD1ItemData` (UPrimaryDataAsset), `FD1InventoryItem` (USTRUCT), `UD1InventoryComponent` (UActorComponent)
  - `ServerUseItem()`, `ServerMoveItem()`, `ServerDiscardItem()` (RPC)
  - 데이터 저장 위치: PlayerState/InventoryComponent 메모리 (세션 종료 시 휘발)
- **Phase 2 (웹서버 연동):** Dedicated Server를 웹서버 Proxy로 두어 HTTP/REST 통신
  - 클라이언트는 절대 웹서버에 직접 접근하지 않음 (보안)
  - Runtime 데이터 (HP, Mana, 쿨타임)는 GAS Replication 그대로 유지
  - 영속 데이터 (레벨, XP, 스킬 해금, 인벤토리)는 로드/세이브 시점에 동기화
- **통신 원칙:** `[Client] ←RPC→ [Dedicated Server] ←HTTP/JSON→ [Web API Server] ←→ [DB]`
